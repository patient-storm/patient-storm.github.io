Project maintenance

This project was to write an application allows users to create and RSVP for upcoming events - essentially a minimal clone of Eventbrite. Here is a description of the code.

Stepping into someone else's application and code is always challenging. Because we used the agile method of development, we did not spend time on unnecesary documentation. More than anything else, we believe in communication. So of course, your best resource is us. You are more than welcome to contact Ayisha Devonish (ayisha.devonish@yale.edu), Dan Levis (dan.levis@yale.edu), Matt Hanna (matt.hanna@yale.edu), and/or Robert Kimball (robert.kimball@yale.edu) for a walkthrough of the application, codes, and its capabilities today. In the event that we are unavailable, however, perhaps the best way to understand the capabilities of the application today is to review the tests that it passes (in tests.js or by running mocha). By reviewing the tests, a new developer stepping into the application will learn what we've built the application to do in its current form. Our sprint reports, which you also can access, have a wealth of information about the development process, issues, solutions, etc.

A few general points about project mechanics. This project largely relies on open source projects of which we suspect you are quite familiar. It is a node.js application, and it uses the following components:

  • express.js: This is a minimal and flexible Node.js web application framework.
  • nunjucks: This is a templating language for JavaScript.
  • morgan: This is a HTTP request logger middleware for node.js. It is written by the same people who write Express.
  • validator: This is a a library of string validators and sanitizers that helps to ensure that parameters sent to our application by users are valid.
  • body-parser: This is a Node.js body parsing middleware. It is also written by the same people who write Express.

A few other general points:

  • The tests run by using mocha (use the command "mocha")
  • Today, the application runs on Heroku
  • You can find our Github repository for the project here.
  • Every page uses the Bootstrap CSS framework
  • Our site has an API at /api/events and is able to retrieve an object with key events containing all events in JSON format

Hope that this is enough to get you going. If not, please do reach out to us in the future.

What we learned about management in general and agile in particular

The last requirement we got from our stakeholders was to do some analysis of how small changes in our website could change user behavior. We decided to answer these strategic questions using A/B testing and Google Analytics. This proved to be a technical challenge for us! One of the biggest issues we encountered was that it was difficult to tell when we had properly implemented the testing code. Our method required bypassing a few of Google Analytics' built in tools, and every test essentially required pushing all the way to production. We were disappointed to see in the end that our test for swapping the word "Donate" for "Support" did not work properly due to bugs in our code. That said, our test for whether adding "RSVP" to our events list influenced behavior worked perfectly.
